Research Plan · URL Token Attribution

Which parts of a URL
actually matter?

A research plan for understanding URL token attribution in query-driven web crawling — what encoder, decoder, and hybrid models attend to when deciding if a link is worth following.

00 · Problem framing

The core question

A query-driven crawler discovers a list of URLs it has not yet fetched. It must order them. The only signal available before fetching is the URL string itself (plus possibly the anchor text of the link that pointed to it). The question is: which tokens within that URL string predict that the target page is relevant to the user's query?

This has two layers — first, building a model that scores URL relevance, and second, explaining which parts of the URL drove that score. The second layer is the novel contribution: no crawling paper has produced per-token attribution over URL strings in a crawl context.

01 · Background to acquire

What to learn before starting

Foundational
URL feature engineering
URL tokenization strategies: delimiter-based (/, ., -, ?, =), WordPiece (BERT-style subword), character n-gram, and hybrid. Understand how each treats e.g. torch.optim.Adam differently. Key papers: Kan & Thi (2005) "Fast Webpage Classification Using URL Features", Baykan et al. (2011).
Foundational
Transformer architectures
Encoder-only (BERT, RoBERTa): bidirectional context, good for classification. Decoder-only (GPT, Llama): autoregressive, can be prompted. Encoder-decoder (T5, BART): sequence-to-sequence. Bi-encoder vs. cross-encoder distinction — critical for latency vs. accuracy tradeoff in URL scoring.
Attribution methods
SHAP for NLP
Partition SHAP and KernelSHAP for text inputs. Shapley values from cooperative game theory — each token's marginal contribution averaged across all coalitions. Key library: shap. For transformers specifically: shap.Explainer with a pipeline. Learn the "sufficiency" and "comprehensiveness" faithfulness criteria.
Attribution methods
Integrated Gradients
Sundararajan et al. (2017) "Axiomatic Attribution for Deep Networks." Integrates the model gradient along a path from a baseline input to the real input. Satisfies completeness axiom. Implemented in captum (PyTorch). Works natively on token embeddings — gives signed attribution per token.
Attribution methods
Attention rollout & gradient×attention
Raw attention weights are not attribution scores — they reflect routing, not causal contribution. Attention rollout (Abnar & Zuidema, 2020) propagates attention across layers. Gradient×attention (Jain & Wallace, 2019) is more faithful. Study both as baselines to compare against IG and SHAP.
Attribution methods
LIME & occlusion
LIME fits a local linear model around a perturbed instance. Occlusion/ablation replaces individual tokens with [MASK] and measures the score drop — the simplest faithfulness test. Both serve as sanity-check baselines for the more sophisticated methods.
IR / crawling
Focused crawling literature
Reviewed: Craw4LLM (2025), Neural Prioritisation ICTIR (2025), Pezzuti et al. (2025). Key concept: harvest ratio, speedup over BFS. Understand why URL-only scoring is appealing — no fetch needed, zero latency overhead.
IR / crawling
Cross-encoder relevance models
MonoBERT, MonoT5, DPR. How [CLS, query, SEP, document, SEP] produces a relevance score. In your setting, "document" = URL string. Fine-tuning on query-URL pairs with binary or graded relevance labels. MS MARCO as the standard fine-tuning dataset.
02 · Related work

What already exists

Crawling systems

These are the practical systems your attribution work would plug into. Understanding their frontier architecture is necessary to motivate why URL-only scoring matters — each system has a priority queue that needs a fast per-URL score before fetching.

Apache Nutch — Cutting & Cafarella (2004)
Apache Foundation · Production web-scale crawler
The canonical open-source crawler. Hadoop-based batch architecture: generate URL list → fetch → parse → update crawlDB. Used by Common Crawl since 2014 to produce the 900M-page datasets used in Craw4LLM. Understands why URL ordering matters: Nutch's generate step selects and scores URLs from the crawlDB before fetching. Your attribution work would inform how that scoring function should be designed.
Heritrix — Mohr et al. (2004), Internet Archive
Internet Archive · Archival-quality crawler
JVM-based, single-machine archival crawler. Strictly respects robots.txt and politeness delays. Used for focused web archiving — exactly the setting where query-driven URL prioritisation matters most. Reference for how frontier management is done in a production system with politeness constraints (the same constraint that makes on-the-fly neural scoring attractive — you score during the wait).
Scrapy — Zyte (2008, open source)
Python ecosystem · Most widely used focused crawler
Python-based async crawler, most popular for focused/domain-specific crawling. The Frontera extension provides a pluggable crawl frontier with priority queues — this is the exact integration point for a URL attribution scorer. Research using Scrapy + Frontera would be reproducible and immediately usable by practitioners.
StormCrawler — DigitalPebble (2014)
Apache Storm · Real-time stream-based crawler
Stream-based rather than batch — URLs are continuously fetched, parsed, and scored in a DAG topology. This real-time model is the one where a fast URL scorer is most critical: there's no batch step to amortise the scoring cost over. Motivates the latency row in the model comparison table.
Crawl4AI — open source (2024)
Python / AI-first · Modern agentic crawler
Modern AI-oriented crawler with a BestFirstCrawlingStrategy that already implements a priority queue scored by a keyword scorer. Your work directly replaces that scorer with a model-backed URL attribution-guided one. Good system to demonstrate the implementation in — it's Python, it's actively maintained, and its scoring API is documented.
Common Crawl
Non-profit · Open web crawl dataset
Produces petabyte-scale open snapshots of the web used in LLM training. Uses harmonic centrality as its URL priority signal. Your work proposes a query-conditioned alternative to this purely graph-based signal. ClueWeb22 (used in Craw4LLM experiments) is a snapshot from a similar commercial crawl — your experimental dataset.

Neural crawling systems (SOTA context)

Craw4LLM — Yu, Liu & Xiong (ACL Findings 2025)
arXiv:2502.13347 · SOTA for LLM-pretraining crawls
Replaces PageRank with a fastText-based pretraining influence scorer as the frontier priority signal. With 21% of URLs crawled, LLMs pretrained on the data match downstream performance of full crawls. Evaluated on ClueWeb22 (900M pages). This is the direct SOTA baseline your crawl evaluation (E5) must beat or match — and the key gap it leaves is that it scores pages by general quality, not by alignment to a specific user query. URL token attribution is exactly what fills that gap.
Neural Prioritisation for Web Crawling — ICTIR 2025
arXiv:2506.16146 · QMin / QFirst policies
Introduces quality propagation over the link graph: score a page's unvisited neighbours using the already-crawled page's LLM quality estimate. QMin (exploit minimum ancestor quality) achieves 1.6× speedup over BFS. QFirst is more exploration-oriented. Key finding: neural estimators identify low-quality pages better than high-quality ones — which is why QMin (avoid worst) beats a "pick best" policy. Your work complements this: rather than propagating quality through the graph, you extract a signal directly from the URL string before any fetch occurs.
Document Quality Scoring for Web Crawling — Pezzuti et al. (2025)
arXiv:2504.11011 · LLM-driven oracle crawling
Establishes a theoretical oracle upper bound on crawl quality by using an LLM to perfectly prioritise pages. Also empirically confirms that high-quality pages tend to link to each other — motivating neighbourhood quality propagation. Important for framing: this oracle requires fetching the page content before scoring it. URL token attribution is a complementary signal that requires no fetch — positioning your contribution as a lightweight pre-fetch filter rather than a replacement for content-based scoring.

URL classification & relevance prediction

Fast Webpage Classification Using URL Features — Kan & Thi (2005)
WWW 2005 · Foundational
First systematic study of URL token sequences as a tree structure for classification. Introduced component-qualified features (token + its URL component origin). Key insight: subdomain and path tokens encode different information. Directly relevant to your tokenization design.
Analyzing URL Structure for Machine Learning — (ISERT 2024)
ISERT 2024 · Feature engineering
Uses SHAP over hand-engineered URL features (host, TLD, path length, token count) to assess feature relevance. Closest existing work to your idea, but uses hand-crafted features rather than token-level attribution from a neural model. Good baseline and framing reference.
Cross-language focused crawling: URL relevance prediction — (2008)
ScienceDirect · Empirical confirmation
Empirically showed that most relevant pages contain the topic token in their URL. Established that URL tokens are a valid predictor of relevance — but used lexical matching, not learned attribution. Your contribution is to replace this with a differentiable model and measure what it learns.

Attribution / interpretability for transformers

Axiomatic Attribution for Deep Networks — Sundararajan et al. (2017)
ICML 2017 · Integrated Gradients
Foundational method. Defines the completeness axiom (attribution scores sum to prediction difference from baseline). The go-to method for token-level attribution in transformers. You'll implement this via captum.attr.IntegratedGradients.
BERT Meets Shapley — Kokalj et al. (EACL 2021)
EACL Hackashop 2021 · SHAP for BERT
Extends SHAP to transformer classifiers. Shows that SHAP applied to BERT token embeddings produces human-aligned explanations. The methodology section is directly transplantable to URL token attribution over a fine-tuned BERT.
Is Attention Explanation? — Jain & Wallace (2019)
NAACL 2019 · Attention skepticism
Critical result: attention weights are not necessarily faithful explanations of model predictions. Motivates using IG and SHAP as primary attribution methods rather than raw attention, with attention rollout as a comparison baseline only.
Quantifying Attention Flow — Abnar & Zuidema (2020)
ACL 2020 · Attention rollout
Proposes attention rollout as a more faithful attention-based attribution by propagating attention across all transformer layers. Use as the attention-based baseline in your method comparison.

Malicious URL detection (adjacent field)

A Survey of Malicious URL Detection — Sahoo et al. (2019)
ACM Computing Surveys · Adjacent
Rich feature taxonomy for URL analysis: lexical features, host-based, content-based. The tokenization strategies (delimiter-based, character n-gram) are directly reusable. Key difference: their task is malicious vs. benign, yours is relevant vs. irrelevant to a query — so the important tokens will differ.
03 · Methodology

How to build it

Step 1 — URL tokenization

URLs are not natural language — tokenization strategy matters. The choice here is WordPiece / BPE: feed the raw URL directly to the pretrained tokenizer of whatever model you're using. This is the only approach needed because you're using off-the-shelf models that already have fixed tokenizers — feeding them a different tokenization would break the embedding space they were trained in.

Strategy — WordPiece / BPE (subword)
Feed raw URL to the model's pretrained tokenizer
The model's own tokenizer (BERT WordPiece, T5 SentencePiece, etc.) processes the URL string directly. Example: pytorch.org/stable/optim → py ##torch . org / stable / op ##tim. This preserves compatibility with the pretrained embedding space — critical since you are not fine-tuning. The subword splits may fragment meaningful slugs (optim → op ##tim), which is the core tension your attribution analysis will surface.

Component annotation: after tokenization, label each token with its structural origin (scheme / host / path / query-param) using a lightweight delimiter parser run before passing to the model. This annotation is used only during attribution aggregation — not as model input — so it doesn't interfere with the pretrained tokenizer.

The interesting research question that follows directly: when the tokenizer fragments a meaningful URL slug into subwords, does the model's attribution still correctly identify the semantic unit (the full slug) as important, or does attribution scatter across the fragments? This is a concrete hypothesis you can test using your synthetic dataset.

Step 2 — Architecture families to compare

You want attribution results across different architectural families to answer: do different architectures agree on which URL tokens matter? Disagreement is itself a finding. Pick one strong representative per family — the specific checkpoint is secondary to the architecture type.

Key framing — no training needed

All families above have off-the-shelf pretrained checkpoints trained on MS MARCO query-passage pairs. You feed them (query, URL string) and extract the score. Your contribution is not building a better scorer — it is auditing what existing scorers attend to inside a URL, across all five architectural families, and comparing whether they agree.

Step 3 — Data: evaluation only, no training split needed

Since there's no training, you don't need labelled data to run the models — they score any (query, URL) pair. Labels are only needed to evaluate whether the scores are actually correct (experiments E1 baseline, E5). The sources below serve evaluation only:

Primary
MS MARCO Passage / Web Search
~8.8M query-passage pairs with URLs, binary relevance. This is also what all the off-the-shelf rerankers were trained on. Use for evaluation — do the existing models' scores correlate with labels when the "passage" is just the URL string? MS MARCO Web Search (2024) adds click labels at web scale.
Evaluation
TREC Web Track
Graded relevance (0–3) for ~50 queries over ClueWeb09/12. Smaller but more carefully judged. Use for held-out evaluation. Graded labels let you measure whether attribution scores correlate with relevance grade, not just binary label.
Controlled
Synthetic controlled dataset
Construct URLs where you know the answer: inject the query keyword into path vs. domain vs. query-param and measure whether the model's attribution correctly identifies the injected token. This is your "ground truth" for attribution faithfulness.
Optional
ClueWeb22 + Craw4LLM setup
900M URLs with quality scores from commercial crawl. Pairs well with the crawl simulation angle — you can extract (URL, quality_score) and treat quality as a relevance proxy. Allows large-scale attribution analysis.

Step 4 — Attribution pipeline

01 Tokenize URL WordPiece via model's own tokenizer. Annotate each token with URL component origin.
02 Load model Off-the-shelf pretrained checkpoint per architecture family. No training loop.
03 Extract score Feed (query, URL) → get relevance logit. Record for evaluation.
04 Run attribution IG via captum. SHAP via shap.Explainer. ColBERT MaxSim natively. Attn rollout.
05 Aggregate scores Map token scores → URL component. Average across queries per component type.
06 Faithfulness test Mask top-k attributed tokens. Measure score drop (sufficiency / comprehensiveness).
07 Crawl eval Use attribution insight as URL ordering signal. Measure harvest ratio vs. baselines.
04 · Experiments & evaluation

What to run

E1
Component-level attribution analysis
RQ: Which URL component (scheme / subdomain / domain / TLD / path / query-param) contributes most to relevance prediction on average?
Run all models on 5K query-URL pairs from MS MARCO. For each URL, compute IG attribution per token. Aggregate scores by component type. Report mean |attribution| per component. Expect domain and first path token to dominate. Compare whether cross-encoder and bi-encoder agree on ranking of components.
mean |attribution| per component rank correlation between models (Spearman ρ) component contribution % pie
E2
Attribution faithfulness evaluation
RQ: Are the attribution scores actually causally related to the model's output, or are they noise?
Two standard tests. Comprehensiveness: mask the top-k highest-attributed tokens and measure how much the relevance score drops — a faithful method should produce a large drop. Sufficiency: keep only the top-k tokens and measure how much score is retained. Compare IG, SHAP, attention rollout, and occlusion across all model types. Use your synthetic dataset as a controlled ground truth — inject query keywords in known positions and verify they receive top attribution.
comprehensiveness@k (score drop) sufficiency@k (score retained) top-k recall on synthetic URLs AOPC (area over perturbation curve)
E3
Query-conditioned attribution shift
RQ: Does the set of important URL tokens change depending on the query?
Hold the URL constant, vary the query. E.g., URL https://pytorch.org/docs/stable/optim.html. For query "Adam optimizer", does optim receive high attribution? For query "PyTorch documentation", does docs dominate instead? This is only observable in the cross-encoder — the bi-encoder will produce query-independent URL attributions. The difference between these is a key finding. Cluster queries by type (navigational / informational / transactional) and test whether attribution patterns differ.
attribution shift Δ across query types cosine similarity of attribution vectors per query cluster qualitative case studies
E4
Subword fragmentation effect on attribution
RQ: When WordPiece splits a meaningful URL slug into subword fragments, does attribution scatter across them or concentrate correctly?
Using the synthetic dataset: construct URLs with a known keyword in a path slug (e.g. /optim/). Run WordPiece tokenization and observe how the slug splits. Then run IG and check whether the summed attribution over all fragments of the slug equals the attribution a hypothetical whole-token slug would receive. Hypothesis: for cross-encoders, attribution concentrates on the fragment containing the keyword root (e.g. op##tim → op carries more). For late-interaction, MaxSim resolves to whichever fragment has the highest embedding similarity to the query token — test whether this is the semantically correct fragment.
attribution mass ratio: correct fragment vs. others top-1 fragment recall on synthetic URLs agreement across architecture families
E5
Attribution-guided URL ordering as a crawl policy
RQ: Does using token attribution scores as a URL priority signal improve crawl efficiency over lexical and graph-based baselines?
Extract the top-k attributed token types from E1 (e.g. "domain + first path token beat everything else"). Build a lightweight scoring function that uses only those token features. Simulate crawls on ClueWeb22 with this as the priority signal. Compare harvest ratio curves against: BFS (baseline), BM25 URL scoring, cross-encoder full URL scoring, and Craw4LLM (SOTA). This closes the loop — attribution insights directly improve the practical crawling system.
harvest ratio @k pages speedup over BFS MRR of first relevant page found latency (ms per URL scored)

05 · Discussion

Nice-to-have in the paper

Fine-tuning as ablation (not main approach)

If time allows, fine-tune one model (e.g. MiniLM cross-encoder) on (query, URL, label) pairs from MS MARCO where the document is the URL string only — not the passage text. Then compare attribution patterns before and after fine-tuning. Does URL-specific training change which tokens the model attends to? This is a clean ablation that adds depth without being the main story.

Attribution methods will likely disagree between encoder and decoder models on which tokens matter. This disagreement is a finding, not a failure — it reveals that different architectures encode URL structure differently, and the crawl policy design should be tailored to the model family.

Practical implication

If E1 shows that domain + first path token explain ~80% of the variance in relevance prediction, you can build a near-zero-cost URL scorer that only needs to parse two tokens. This is a deployable insight for production crawlers that need to score millions of URLs per second.

Attribution method robustness

Discuss the attention-is-not-explanation problem explicitly. Show the discrepancy between attention rollout and IG/SHAP on the same URLs. This grounds the choice of IG/SHAP as primary methods and will be necessary to preempt reviewer pushback.

Open questions worth raising

  • Dynamic URLs: Many modern web apps generate URLs with UUIDs, session tokens, and encoded parameters. Attribution scores on these are meaningless. How should a system handle or filter them? This is an unsolved practical problem.
  • Query type sensitivity: Navigational queries (where the user wants a specific site) will heavily attribute the domain token. Informational queries will attribute path tokens. Does a single model handle both, or should query-type routing be added?
  • Multilingual URLs: Punycode-encoded domains, Chinese Pinyin paths, Arabic transliterations. Attribution over multilingual URL tokens is almost completely unstudied. Even just raising this as a limitation is a solid contribution.
  • Anchor text as a bridge: The link that discovered the URL often has anchor text. Attribution over (anchor text + URL) jointly — what's the interaction? A future direction.
  • Does more attribution accuracy mean better crawl policy? There's a potential decoupling — a slightly less faithful attribution method might still produce a better ordering signal. Discuss whether attribution evaluation and crawl evaluation should be treated as separate objectives.

Societal / ethical angle

URL token attribution could be used adversarially — domain owners could learn which URL patterns get crawled first and structure their URLs to exploit that. This is the URL equivalent of keyword stuffing. Worth a brief mention for completeness.
